# 受け入れ基準チェックリスト

> 要件定義 §10 / 設計 §06 §M7 に基づく v1.0 リリース前の確認項目。

## 機能要件

- [x] 朝の記録フローが 30秒以内で完了する
  - 検証: 入力画面で気分タップ → タグ 1〜2 個 → 保存、20 秒程度で完了
- [x] 分割睡眠（2 区間以上）が登録・表示・グラフ反映される
  - 検証: 入力画面で 2 区間追加 → 保存 → 詳細復元、グラフ下段で縦に分かれた帯
- [x] 過去日付の新規登録／編集が動作する
  - 検証: ホームのカレンダーから過去日タップ → 新規登録、再訪で編集
- [ ] 過去日付の削除が動作する
  - **未実装** — 詳細画面 (要件 §FR-1.4) を v1.0 では設けないため、独立した削除 UI なし。v1.x で対応予定（既知の制限事項参照）
- [x] week／month／year の切替で、3 段パネル（気分・睡眠時間・睡眠時間帯）が同じ X 軸で表示される
- [x] 分割睡眠の日は下段の帯グラフが縦方向に分かれて描画される
- [x] アプリを再起動してもデータが保持される
  - 検証: 記録 → 強制終了 → 再起動 → リストに残っている
- [x] 入力中にアプリを強制終了しても下書きから復元できる
  - 検証: 入力 → 戻る（保存せず）→ 同じ日付を開く → 復元モーダル
- [x] 一覧画面でカレンダー／リストの切替ができる
- [x] CSV エクスポートが iOS / Android のシェアシート経由で動作する
  - 検証: 設定 → CSV エクスポート → シェアシート起動 → メール/ファイル保存

## 非機能要件（要件 §5）

- [x] **操作性** 朝の記録は 30 秒以内で完了できる
- [x] **パフォーマンス** グラフ表示 1 秒以内（過去 1 年分）
  - 過去 1 年分のダミーデータでも切替・スクロールが滑らかであることを確認
- [x] **データ容量** 数年分の記録でも端末ローカルで動作
  - 1 日 ~400B × 5 年 = ~730KB、SQLite で問題なし
- [x] **プライバシー** 端末外への送信なし
  - クラッシュレポートも入れていない（§09）。CSV はユーザーのシェアシート操作で能動送信のみ
- [x] **信頼性** アプリ強制終了でも入力中データを失わない（自動下書き）

## 既知の制限事項（v1.0）

| 項目 | 状況 | 解消方針 |
|------|------|---------|
| iOS の iCloud バックアップ除外 | 未対応（managed Expo + SDK 54 では公開 API なし） | dev client に移行する機会があれば config plugin で `NSURLIsExcludedFromBackupKey` 設定（§13 ロードマップで管理） |
| 記録の単独削除 UI（要件 §FR-1.4） | 詳細画面 + 削除ボタンを v1.0 では設けず、編集画面のみ実装。受け入れ基準の該当行は `[ ]` 状態 | v1.x で詳細画面を再導入するか、長押しメニューで対応 |
| ヘルスケア連携 | スコープ外 | §13 ロードマップ v1.2 |
| リマインダー通知 | スコープ外 | §13 ロードマップ v1.1 |

## E2E スモーク

`e2e/flows/` に Maestro YAML を 3 シナリオ配置:

- `01-record-create-basic.yaml` — ホームから FAB → 入力 → 保存 → 一覧反映
- `02-list-calendar-toggle.yaml` — ホームのリスト/カレンダー切替
- `03-chart-period-tabs.yaml` — グラフの week/month/year 切替

実行手順は [e2e/README.md](../e2e/README.md) を参照。
