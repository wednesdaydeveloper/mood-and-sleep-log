# v1.4 日記項目

> 記録入力画面のメモの **直後** に「日記（任意）」複数行テキストエリアを追加し、長文の記録（最大 5000 文字）を構造化して保存・検索できるようにする。

## 1. 機能要件

### 1.1 入力 UI（記録入力画面 §FR-1.8）

- **メモ欄の直後**に「📓 日記（任意）」セクションを追加
- 複数行 TextInput（multiline）
- 文字数上限: **5000 文字**
- プレースホルダ: `その日あったこと、思ったことなど`
- 空欄入力は内部的に `null` に正規化

### 1.2 永続化

- `daily_record` テーブルに 1 カラムを追加
  - `diary TEXT`（nullable）
- 既存レコードはマイグレーション直後すべて `null`

### 1.3 検索

- ホームリストのキーワード検索の対象に **diary を追加**
- 既存対象: memo / 感情タグ / event
- 新対象: memo / 感情タグ / event / **diary** （OR 条件、大文字小文字区別なし）

### 1.4 リスト表示

- ホームリストには **表示しない**（リスト膨張防止、長文は詳細画面で確認）
- 検索ヒットは可能（表示はされないが該当行が結果に含まれる）

### 1.5 グラフ

- データポイントポップアップには **表示しない**（長文不向き）

### 1.6 CSV

- エクスポート列に **`diary`** を追加（9 列形式）
- インポート時の互換性:
  - **v1.4** (9 列) → そのまま読込
  - **v1.3** (8 列、diary なし) → diary を null として読込
  - **v1.2** (7 列、event/diary なし) → 両方 null として読込
  - **v1.0/v1.1** (5 列、服薬・event・diary なし) → 全て null として読込

## 2. スコープ外（v1.4 では実装しない）

- リスト表示・カレンダーセル表示
- グラフポップアップ表示
- 過去データの自動移行
- リッチテキスト（マークダウン等）
- 画像添付
- 画面内文字数カウンタ表示（`maxLength` で上限到達時の入力停止のみ）

`DEFAULT_FORM_VALUES.diary = null`。

### 3.3 DB スキーマ（Drizzle）

```typescript
// src/db/schema.ts に追加

  // 既存カラム ...
  diary: text('diary'),  // nullable, max 5000 chars はフォーム層で保証

```

### 3.4 マイグレーション

- `0003_gigantic_gwen_stacy.sql`:

### 5.2 upsert / replaceAll

- 既存の INSERT / UPDATE 文に diary カラムを追加

## 6. CSV 連携

### 6.1 エクスポート（recordsToCsv）

- ヘッダー末尾に `diary` を追加
- 値は文字列、null は空文字
- 改行を含む場合は通常の RFC 4180 escape（フィールド全体をクォート、内部の `"` は `""` でエスケープ）

### 6.2 インポート（parseCsv）

- ヘッダー検出を 4 種に分岐:
  - `current` (9 列、event + diary)
  - `legacy_v13` (8 列、event のみ)
  - `legacy_v12` (7 列、服薬まで)
  - `legacy_v11` (5 列、最小)
- 各形式で `expectedColumns` を切り替えて parseRow を実行
- 旧形式では diary 欄なし → null

### 6.3 後方互換性

| CSV 形式 | バージョン | 列数 | diary 取込結果 |
|---------|----------|------|--------------|
| current | v1.4+ | 9 | そのまま |
| legacy_v13 | v1.3 | 8 | null |
| legacy_v12 | v1.2 | 7 | null |
| legacy_v11 | v1.0/v1.1 | 5 | null |

## 7. Excel 変換ツール（convert/convert.py）

- 元 Excel に「日記」列が存在し、Dropbox URL や長文が入っているケース
- v1.3 までは無視していたが、v1.4 から **専用 diary 列に出力**
- 5000 文字以内に切り詰め（超過は末尾を切る）

- 旧形式 CSV のインポートが引き続き動作する

## 10. リスク・注意

| リスク | 対応 |
|--------|------|
| マイグレーション失敗で既存データ破損 | `ADD COLUMN` は追加のみ、既存データに影響しない |
| 旧形式 CSV のインポート失敗 | ヘッダー検出を 4 種に分岐させ、列数も動的に切替 |
| 長文 diary で UI もたつき | TextInput multiline は標準実装、5000 文字程度なら問題なし |
| CSV 内の改行で行が破損 | RFC 4180 のクォート＋エスケープで保護（既存の parseRows がフィールド内改行を許容） |
| 端末ストレージ圧迫 | 5000 文字 × 365 日 ≒ 1.8MB/年。SQLite 的には誤差 |
