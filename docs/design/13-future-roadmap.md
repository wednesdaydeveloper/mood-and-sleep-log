# 将来ロードマップ

> 要件定義 §8 のスコープ外項目を将来どう実現するか、現設計への影響を整理。

## 1. 優先度マトリクス

| 機能 | ユーザー価値 | 実装難度 | 優先度 |
|------|-------------|---------|-------|
| リマインダー通知 | 高（記録忘れ防止） | 低 | ★★★ |
| ヘルスケア連携（自動取得） | 高（手入力削減） | 高 | ★★★ |
| AI 傾向分析・コメント | 中 | 中〜高 | ★★ |
| クラウド同期 | 中（複数端末） | 高 | ★★ |
| ウェアラブル連携 | 中 | 高 | ★ |
| マルチユーザー（家族共有等） | 低（自分用前提） | 高 | ★ |
| 感情タグのカスタマイズ | 中 | 低 | ★★ |
| データインポート（CSV） | 中（移行需要） | 中 | ★★ |
| **DB 暗号化（SQLCipher）** | 中（健康データ保護） | 中（dev client 必須） | ★★ |

★★★ = v1.x で検討、★★ = v2.0、★ = v2.x 以降

## 2. リマインダー通知（v1.1 想定）

### 機能
- 毎朝設定した時刻にローカル通知「昨日の記録を入力しましょう」
- 通知タップで記録入力画面（昨日）が開く

### 設計影響
- `expo-notifications` 追加
- 設定画面に「通知時刻」項目
- 通知許可ダイアログ：初回起動時 or 設定タップ時

### 既設計との整合
- 通知タップで `record/[date]` に直接遷移（expo-router の deep link 対応で可能）
- 設計01〜06には影響なし

## 3. ヘルスケア連携（v1.2 想定）

### 機能
- iOS HealthKit / Google Fit から **睡眠データを自動取得** → タイムラインUIに初期値として反映
- ユーザーは確認・微調整して保存

### 設計影響
- `react-native-health` (iOS) / `react-native-google-fit` (Android) 追加
- 入力画面に「ヘルスデータから取り込む」ボタン
- 権限要求（プライバシーへの考慮、§11）

### 既設計との整合
- `SleepInterval` のデータモデルはそのまま流用可能
- タイムラインUIは既存のドラッグ操作で微調整可能 → そのまま使える
- 重要：**インポートしたデータの出所を区別**（`source: "manual" | "healthkit" | "googlefit"` を追加検討）

## 4. AI 傾向分析（v2.0 想定）

### 機能例
- 「最近の気分が下がり気味です。睡眠時間も短いようです」
- 月次レポート（気分と睡眠の相関、改善傾向）

### 実装方針
- **オンデバイスAI**（プライバシー優先）：Core ML / TensorFlow Lite
- または **クラウドLLM**（API 経由、データ匿名化）

### 設計影響
- データを匿名化する処理層が必要（メモは送信しない、タグは集計値のみ）
- §11 セキュリティ・プライバシーの再検討必須

## 5. クラウド同期（v2.0 想定）

### 機能
- 複数端末でデータ共有
- バックアップ復元

### 候補
- **Supabase**（PostgreSQL + Auth、無料枠あり）
- **CloudKit**（iOS のみ、無料）
- **Firebase**（Google）

### 設計影響大
- 認証システム追加（メアド + パスワード、または Apple/Google ログイン）
- データモデルに `userId` を追加
- コンフリクト解決（同じ日付を複数端末で編集）
- 既設計のローカルDB方針と並行して動かす必要

設計影響が大きいため **v1 はローカルのみで完成度を上げる**、同期は別途プロジェクト的に切り直す。

## 6. ウェアラブル連携（v2.x）

- Apple Watch / Wear OS で気分の超軽量入力
- 通知タップで5タップ以内に記録完了

設計影響: 大（OS連携、別アプリ実装）

## 7. 感情タグのカスタマイズ（v1.x）

- ユーザーが独自タグを追加・削除・並び替え

### 設計影響
- 現在: タグはコード内のプリセット定数
- 将来: タグも DB テーブル化
  ```sql
  CREATE TABLE tag (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    category TEXT,
    sortOrder INTEGER,
    isDefault INTEGER  -- 1: 初期データ、0: ユーザー追加
  );
  ```
- マイグレーションで初期プリセットを INSERT

将来追加するなら、現状から **テーブル化を先にやる選択** もある（M3 までに判断）。

## 8. DB 暗号化（v2.x 想定）

### 機能
- SQLCipher による DB ファイル暗号化
- 端末紛失時の保護を強化

### 実装方針
- `expo-sqlite` から `op-sqlite + SQLCipher` へ切り替え
- 暗号鍵は `expo-secure-store`（iOS Keychain / Android Keystore）に256bit ランダム鍵を保存
- 既存の plain DB から暗号化 DB へのデータ移行スクリプトが必要

### 設計影響
- **dev client への移行が必須**（Expo Go では動かない）
- 起動時の DB オープンが数百ms 遅くなる
- 依存パッケージが増える

### トリガ
- ストア公開する場合
- マルチユーザー対応で他人のデータを扱う場合
- v2.x で dev client 化する別の理由ができた場合

## 9. データインポート（v1.x〜v2.0）

- CSV / JSON のインポート
- バックアップから復元
- 他アプリからの移行

### 設計影響
- パーサ実装（CSVは出力と同フォーマットを受入）
- 重複検出（既存日付は上書き / スキップ / マージ選択）
- バリデーションエラー時の挙動

## 11. 設計上、今のうちに気をつけておくこと

実装段階で **将来を縛らないよう** 意識する点:

| 項目 | 注意 |
|------|------|
| データモデル | `source` フィールド追加余地（HealthKit 連携時） |
| 感情タグ | プリセット直書きより、定数モジュールで一箇所管理 |
| 同期キー | 主キーは UUID で生成（クラウド同期時に衝突しにくい） |
| 時刻 | 全て UTC unix秒で保存（タイムゾーン跨ぎに強い） |
| API 層の抽象化 | Repository で抽象化（将来クラウドへ差し替え可能） |
| UIテキスト | i18next 経由（§12） |

これらは現設計に既に組み込み済み。
