# セキュリティ・プライバシー設計

## 1. 前提

- **完全オフライン動作**：端末外への自動送信は一切なし
- **シングルユーザー（自分専用）**：認証システムなし
- 取り扱うデータは健康情報・気分という **センシティブな個人情報**

## 2. データ保護方針

### 端末内ストレージ

| 項目 | 内容 |
|------|------|
| 保存場所 | アプリのサンドボックス内 SQLite ファイル（plain `expo-sqlite`） |
| 暗号化 | OS 標準のサンドボックス保護のみ（DB レベルの暗号化は不採用） |

### DB 暗号化を不採用とした理由

当初は SQLCipher 採用を検討したが、以下の理由で **plain `expo-sqlite`** を採用する:

- **Expo Go との非互換**: SQLCipher（`op-sqlite` 経由）は Expo development build (dev client) が必須で、Expo Go で動かない
- **開発フローの維持**: M2〜M6 の UI 実装フェーズで Expo Go の手軽な動作確認ループを保つ
- **リスク評価**: 自分専用アプリで端末紛失時の脅威モデルは限定的、OS のサンドボックス保護で許容範囲

### OS 標準の保護で得られる防御

- **iOS**: アプリサンドボックス + Data Protection（端末ロック中はファイル暗号化）
- **Android**: 内部ストレージ + アプリサンドボックス（root 化されていない端末では他アプリからアクセス不可）
- アプリアンインストールで全データ消去

### 将来の暗号化導入余地

dev client へ移行する機会があれば `op-sqlite + SQLCipher` への切り替えは可能。鍵管理は `expo-secure-store`（iOS Keychain / Android Keystore）。§13 ロードマップに「DB 暗号化（v2.x）」として記載。

## 3. アプリロック

**今回は実装しない**（v1 スコープ外）。

理由: 自分専用で頻繁起動するためロックがストレスになる。OS 標準のサンドボックス保護で最低限のアクセス制限は確保される。

将来必要になれば `expo-local-authentication` で追加（§13 ロードマップに含める）。

## 4. バックアップ・データ転送

### 方針: クラウドバックアップから除外（確定）

プライバシー優先。端末紛失時の復旧手段は **CSV エクスポート** に統一。

### iOS
- iOS の iCloud バックアップ除外には `NSURLIsExcludedFromBackupKey` フラグが必要だが、expo-file-system v19 で公開APIから外れたため、ネイティブ呼び出しが必要
- **当面の対応**: SQLite はデフォルトの `Library/SQLite/` に置かれる（バックアップ対象）。M7（仕上げ）で expo-modules-core 経由のネイティブ呼び出し or app.json の `infoPlist`/Entitlements 設定で除外を実装する

### Android
- `AndroidManifest.xml` で `android:allowBackup="false"` を設定
- `android:fullBackupContent` のホワイトリストにも DB を含めない

### 端末交換時の移行手順（READMEに明記）
1. 旧端末: 設定 → CSV エクスポート → クラウドストレージや AirDrop で転送
2. 新端末: アプリインストール後、CSVインポート機能で取り込み（v1.x で実装、§13）
3. v1.0 の段階では **手動再入力** がフォールバック

## 5. CSV エクスポート時のリスク

- CSV にはメモ含む全データが含まれる
- シェアシート経由で **任意のアプリ** に渡せる（メール、クラウド等）
- ユーザー責任で扱う：UIに注意書きを出す

設定画面の CSV エクスポートボタン下に:
> ⚠️ エクスポートしたデータには気分・メモ等が含まれます。共有先にご注意ください。

## 6. ログのプライバシー

- ログ（§09）に **メモ本文を含めない**
- タグ名は集計には使うが、ログにはハッシュまたは件数のみ
- スタックトレースは個人情報を含まないことを意識

```typescript
// ❌ 悪い例
logger.log("info", "save", "saved", { memo: data.memo });

// ✅ 良い例
logger.log("info", "save", "saved", { hasMemo: !!data.memo, tagCount: data.moodTags.length });
```

## 7. 第三者送信

- **第三者への送信は一切なし**（クラッシュレポート Sentry も §09 でオフと判断）
- 将来 Sentry を入れる場合は:
  - メモ本文・タグ名のスクラビング
  - ユーザーオプトイン

## 8. 権限要求

| 権限 | 用途 | タイミング |
|------|------|-----------|
| なし | DBはサンドボックス内のみ | - |
| なし | ファイル共有は標準シェアシート経由（権限不要） | - |
| (任意) Local Authentication | アプリロック有効時のみ | 設定でON時 |

カメラ、位置情報、通知などは要求しない。

## 9. 依存ライブラリの管理

- `npm audit` を定期実行（CIに組込）
- Renovate / Dependabot で脆弱性アップデート通知
- メジャーバージョンアップは慎重に（Expo SDK のリリースノート確認）

## 10. ソースコードのシークレット管理

- 自分専用なのでサーバーAPIキー等はないが:
- Expo の `extra` を `eas.json` で管理、コミット禁止のものは `.env` + `.gitignore`
- Sentry DSN を入れる場合は EAS Secrets に保存

## 11. プライバシーポリシー

- ストア公開しないため不要
- ただし README に「データは端末内のみ、外部送信なし」を明記

## 12. 確定事項

- ✅ **DB 暗号化なし**（plain `expo-sqlite`、OS 標準のサンドボックス保護に依存）
- ✅ **アプリロックなし**（v1 スコープ外）
- ✅ **クラウドバックアップから除外**（移行は CSV エクスポート経由）
- 🔮 **将来**: dev client 移行と一緒に SQLCipher 導入を検討（v2.x）
